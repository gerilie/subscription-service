# 🤝 Contributing to Subscription Service

Спасибо, что решил помочь проекту! Я ценю любые улучшения — от исправления опечаток до новых фич.

## 🚀 Быстрый старт

1. **Форкни репозиторий** на GitHub.

2. **Склонируй репозиторий** локально:

   ```bash
   git clone https://github.com/your-username/subscription-service.git
   cd subscription-service
   ```

3. **Подними окружение**:

   ```bash
   make dev-up
   ```

4. **Проверь, что всё работает**:

   ```bash
   make test
   ```

5. **Открой Swagger**:

   ```
   http://localhost:3000/swagger/index.html
   ```

## 📦 Структура проекта

```text
.
├── cmd/subscription/ # Точка входа (main.go)

│   ├── config/       # Конфиг приложения
│   └── subscription/ # Вся логика сервиса
├── migrations/       # SQL-миграции (Goose)
├── docs/             # Swagger-документация
├── Makefile          # Команды сборки, тестирования и разработки
```

## 🧪 Как написать тесты

Все тесты используют пакет `stretchr/testify`.

### Юнит-тесты

```bash
make test-unit
```

### Интеграционные тесты

```bash
make test-integration
```

### Все тесты

```bash
make test
```

### Правила

* Каждый новый сервис или обработчик должен быть покрыт тестами.
* Для изменений, связанных с БД, обязательны интеграционные тесты.
* Код покрывать black-box тестами.

## 📝 Как добавить новую фичу

1. Создай новую ветку:

   ```bash
   git checkout -b feature/your-feature-name
   ```

2. Напиши код.

3. Следуй принципам проекта:

   * Чистая архитектура (`Handler → Service → Repository`)
   * Логирование через `pkg/logger`
   * Обработка ошибок через `fmt.Errorf(... %w ...)` или `Sentinel Errors`

4. Если изменяется схема БД, создай миграцию:

   ```bash
   make migrate-gen name=your_migration_name
   ```

5. Если изменился API, обнови Swagger:

   ```bash
   make swagger
   ```

6. Запусти линтер:

   ```bash
   make lint
   ```

7. Напиши тесты.

8. Сделай коммит:

   ```bash
   git add .
   git commit -m "feat: your feature description"
   ```

9. Отправь изменения:

   ```bash
   git push origin feature/your-feature-name
   ```

10. Создай Pull Request на GitHub.

---

## 🔍 Что проверяется в Pull Request

Перед созданием PR убедись, что:

* ✅ Все тесты проходят (`make test`)
* ✅ Линтер не сообщает об ошибках (`make lint`)
* ✅ Swagger обновлён (если менялся API)
* ✅ Новая логика покрыта тестами
* ✅ Нет мёртвого кода и неиспользуемых переменных
* ✅ Ветка актуальна относительно `main` и не содержит конфликтов

## 🧹 Code Style

* Используй `gofmt` для форматирования кода.

* Переменные называй в `camelCase`, экспортируемые сущности — в `PascalCase`.

* Оборачивай ошибки контекстом:

  ```go
  fmt.Errorf("failed to create subscription: %w", err)
  ```

---

## 🐛 Сообщения об ошибках

Если ты нашёл баг, создай **Issue**, указав:

* шаги для воспроизведения;
* ожидаемое поведение;
* фактическое поведение;
* логи или скриншоты (если есть).

---

## 💡 Идеи для улучшений

Если у тебя есть предложение по улучшению проекта, создай **Issue** с меткой `enhancement`.

Не стесняйся предлагать даже крупные изменения, например:

* переход на другую БД;
* новый middleware;
* изменение архитектуры;
* улучшение производительности.
